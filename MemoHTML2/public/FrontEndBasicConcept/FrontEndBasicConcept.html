SPA(Single Page Application)

-단순하게 한개의 페이지만 있는 어플리케이션 이라 정의하며, SPA는 한개의 HTML 파일에서 필요한 데이터만 호출하여 화면을 새로 구성해주는 것이다
-처음에 접속시 한번만에 요청되고 페이지를 이동할때 모든 부분의 코드생성,랜더링을 하지 않고 view부분만 받아서 랜더링하기 때문에 랜더링 속도가 빠르다
-페이지별 HTML파일이 존재하지 않고 JSON,XML으로 페이지 구성되어 있다
-최초 한번 페이지 전체를 로드하고 이후 부터는 특정 부분만 Ajax을 통해 데이터 바인딩하는 방식
-SPA의 중요한 핵심은 웹(UX)의 사용성증가,부가적으로 속도 향상도 얻음 >> 모바일 환경에서는 트래픽 최소화 , 속도 반응성 ,사용성 이슈인데 SPA의 구조도 비슷한구조이다

  • 단점

    -초기 구동 시간이 걸림
    모바일의 네이티브 앱 사용할시 다운후에 여러 동기화등을 거쳐서 사용한다. 그후도 사용 할때 약간의 시간이 지나고 사용할수 있는것을 생각해보면
    SPA의 근본은 모바일 네이티브 앱과 같은 효과를 웹에서 제공하려 한다. 즉 앱의 사용성을 추구하고자 SPA에서도 초기에 필요한 리소스를 다운한다
    초기 구동 시간을 손해 보고 전체적인 앱 속도,반응성,사용성추구 하기 위해서 이다
    이런점을 해결하기위해서 리소스를 청크 다위로 묶어서 해당 리소스에 대한 요청이 있을때만 다운로드를 받는 방법을 적용하면 조금이나마 해결 가능!!
    청크란 하나의 덩어리라는 뜻으로, 코드 스플리팅 시 생성되는 자바스크립트 파일 조각을 의미합니다. 대표적인 방법은로 WebPack 프레임 워크로 Next.js
    codeSpliting 관련 포스팅

    -검색엔진(SEO) 색인화 어려움
    SPA는 위에 설명한 것처럼 예를들면 모바일 앱에 비유를 했고 모바일 앱들은 SEO와 같은 문제를 신경쓰지 않아서 이런문제가 발생하는것이다

SSR VS CSR

  • SSR(Server Side Rendering)전통적인 웹방식

    브라우저에 나타나는 형태 그대로를 HTML로 만들어 제공하고, 브라우저는 HTML을 표시하는 방식이었다.
    요청마다 새로고침이 일어 나며 서버에 새로운 페이지에 대한 요청을 하는 방식
    서버 측에서 HTML & VIEW를 결정해서 줌 >> Blog페이지 생각하면 HTML파일을 express로 인해 주고 받음(ejs,pug와 같은 templete engine들도 있음)

    • 장점

      SEO가능

  • CSR(Client Side Rendering)전통적인 웹방식

    보통 SPA 구동 방식에서 많이 볼수 있고 언제 새로운 데이터를 불러와야할지 스스로 구현 필요
    HTML문서 교환이 아니 필요한 데이터 JSON,XML과 같은 데이터 만 요청을 한다

    • 장점

      트래픽 감소 >> 필요, 변경된 데이터만 받아옴
      사용자 경험 >> 새로 고침 하지 않아 사용자가 네이티브 앱과 비슷한 경험가짐

    • 단점

      SEO문제
      자바 스크립트 위주로 돌아가는 프로젝트는 자바스크립트 엔진이 돌아가징 낳은면 원하는 정보를 표시하지못함
      구글은 검색 엔진은 자바스크립트 엔진이 내장되어 있어 크롤링되지만 네이버,다음의 검색 엔진은 내장되어 있지않음
