  • 안드로이드에서 기본으로 제공하는 4개의 컴포넌트

    • 액티비티(Activity)

      모든 View배치 되는곳이며 실제 프로젝트를 만들면 가장 메인 클래스인 Activity를 제공받아 상속 받는 모습을 봄

      View나 UI를 배치함

      Activity는 생명주기를 가짐

    • 서비스(Service)

      서비스는 눈에는 보이지 않지만 백그라운드에서 구동되는 컴포넌트

      백그라운드에서 실행될때 마다 앱은 기기의 제란된 리소스(RAM)를 사용하며 과하게 사용시 사용자 환경이 손상될 수 있으니 주의

      ex) 스마트폰 사용시 음악 앱으로 노래를 켠다음 앱을 종료해도 음악이 들리는데 이와 같은 방법은 서비스를 이용하여 플레이 해주는 부분을 구현한것이다

      Service 또한 생명주기를 가짐

    • 브로드 케스트 리시버(Broadcast Receiver)

      번역시 방송 수신자로 방송국에서 TV프로그램을 방송한다 그리고 그 TV프로그램을 우리들은 시청이 가능하며 사용자는 시청에대한 의사를 표현한다 >> 여기서 방송국은 불특정 사용자에게 송출하며 TV프로그램을 시청하기 원하는 사람은 시청하는데 >>시청하기 원하는 이 바로 브로드 케스트 리시버 이다

      ex)안드로이드 OS는 문자 메시지가 오면 모든앱에 문자 왔다가 알림이 오는데 메세지를 받기 워해서 브로드 캐스트 리시버를 설정하여 문자가 왔을때 이벤트를 처리하면된디

    • 콘텐츠 프로바이더(Contents Provider)

      번역시 데이터를 제공하는것을 의미함

      ex)전화번호부 앱이다. 카카오톡이나 페이스북에서 설치하면 자동으로 친구 추가가 되는이유 이며 이 전화번호부 앱에는 자신의 DB를 가져다 쓰라는 콘텐츠 프로바이더를 제공합니다

      카카오톡이나 페이스북과 같은 앱들은 콘텐츠 리솔버(Contents Resolver)를 가지고 콘텐츠 프로바이더가 제공하는 DB에 접근하여 읽어 올수 있다

    위 사진과 같이 액티비티 호출하게 되면 액티비티가 화면에 뜸과 동시에 배치된 각종 뷰들이 보임

    미리 알고 가는 뷰,뷰그룹,레이아웃 등등

    • View
      -안드로이드 사용자 인터페이스의 모든 요소는 View 클래스(android.view.View)의 서브 클래스로 . 안드로이드 SDK에서는 사용자 인터페이스를 구성하는데 있어 여러뷰 클래스를 제공한다
      EX)Button , CheckBox , TextView
      -이런 뷰들은 위젯,컴포넌트라고 부른다.
    • ViewGroup
      -여러 개의 다른뷰로 구성될 수 있는 복합뷰(Composite view)라고 하며 이런 뷰를 안드로이드 ViewGroup 클래스(android.view.ViewGroup)로 부터 상속받는다>>ViewGrouped 또한 View의 서브 클래스다
      -예를 들면 RadioGroup이 있다. 이것은 여러 개의 RadioButton 객체들을 포함하여 항상 하나의 버튼만이 선택될 수 있게 한 것이다. 구조적인 관점에서 복합 뷰는 다른 뷰(자식 뷰(child view))들을 포함할 수 있는 하나의 부모 뷰(parent view)로 구성된다. // RadioGroup이 RadioButton 객체들을 포함하듯이.
      -그 부모 뷰는 ViewGroup 클래스로부터 상속받으며, 컨테이너 뷰(container view) 또는 루트 요소(root element)라고도 한다. 또 다른 ViewGroup 기반 컨테이너 뷰로 레이아웃 매니저(layout manager)가 있다.
    • Layout-레이아웃(Layout)이라고 하는 뷰들을 제공하며 레이아웃은 컨테이너 뷰이므로 ViewGroup의 서브 클래스이며 자식뷰들은 화면에 위치시키는 방법을 제어하는 목적으로 설계 되었다

    • View 상속트리

    • 뷰 계층구조
      각 뷰는 화면의 직사각형영역을 가지며 직사각형 안에 그려지는 것에 대한 책임 , 회면의 뷰 영역에 발생하는 이벤트에 응답해야함

      사용자 인터 페이스 화면은 뷰 계층구조로 구성되며 뷰 계층 구조는 계층구조트리의 맨위에 위치한 루트(root)뷰와 그밑의 가지에위한 뷰로 되어 있다.

      컨테이너 뷰의 자식 뷰는 자신의 부모 뷰 위에 포개져서 화면에 나타나되, 부모 뷰의 화면 영역 경계 안에만으로 그 범위가 제한된다

    onCreate >> onStart >> onResume >> onPause >> onStop >> onDestory

    callBack method를 순서로 맞게 나열했음

    1. onCreate
      -클래스에서 생성자라고 말할 수 있으며 액티비티가 실행되면 최초에 한번 실행되고 특수한 경우가 아니면 실행되지 않는다
      -onCreate는 각종View들의 초기화, 각종 데이터들의 초기화가 이루어짐
    2. onStart
      -실용성이 많이 떨어지며 액티비티가 onStop 되었다가 다시 화면으로 돌아 올때 데이터 변경해주어야 할부분 코딩
      -onRestart와 다른것은??
      -onStart는 액티비티가 실행될때 한번 실행된다는 점
      -onRestart는 무조건 onStop되었다가 액티비티가 화면으로 다시 돌아왔을때 실행됨
    3. onResume
      -onResume은 실제로 자주 사용하며 모든일을 여기에 프로그래밍한다
      -통신, 스레드들 실행시킨다
    4. onPause
      -액티비티가 포커스를 잃어버렸을때 호출됨(팝업창이 뛰어졌을때 해당 액티비티위에 팝업이 떠있어서 사용자의 포커스가 팝업에 있어 액티비티가 흐려지는 상황)
      -이때 onResume에서 했던 작업들을 저장하거나 멈추게 하는 코드를 넣는다
    5. onStop
      -홈키를 눌러 백그라운드에 있을때나 다른앱이나 액티비티가 상단에 올라왔을때 내액티비티가 아무것도 안보이게 된 상태에서 실행
      -onPause >> onStop 순으로 실행됨
      -메모리가 부족하여 액티비티가 강제 종료될수 있음
      -모바일 메모리가 한정적이기 때문에 부족할시 , 안쓰는 액티비티를 종료하여 메모리를 확보하는 정책 사용
      -만일 액티비티가 강제 종료 되면 다시 액티비티로 돌아 왔을때 이런데이터를 바탕으로 onCreate를 실행
      -ex)앱사용하다 전화를 받고 통화 후에 앱을 다시 봤는데 메모리가 부족하여 액티비티 날리는 상황을 방지하기 위해서 onStop메소드에서 데이터 백업 실시
    6. onDestory
      -액티비티가 완전히 종료되면 호출된다 -이때 사용하던 메모리 정리하거나 메모리 해제(release)를 한다
